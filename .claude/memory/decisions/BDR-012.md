# BDR-012 — `sync-matches.ts` : mapping team ID inline, garde anti-null, scope manuel

| Décision                                                                                                                                                                    | Pourquoi                                                                                                                                                                                                                                                                                | Alternatives considérées                                                                                                                                         | Statut |
| --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ |
| Le mapping club interne (slug) ↔ team ID football-data.org vit **inline dans `scripts/sync-matches.ts`**, pas dans `src/data/` ni dans le type `Club`                       | C'est un détail d'implémentation propre à cette source de données, pas une donnée de domaine consommée par l'app — cohérent avec [BDR-001](BDR-001.md) (pas d'abstraction avant besoin réel ailleurs)                                                                                   | Champ `footballDataOrgId` dans `Club`/`clubs.json` (écarté : couple le type de domaine à un fournisseur externe, pour un besoin qui n'existe que côté script)    | actif  |
| Le script applique une garde anti-régression : `scoreFinal = scoreFetché ?? scoreExistant` (jamais écraser un score connu par un `null` renvoyé par l'API)                  | Reprend la règle déjà actée au niveau projet (table de décisions du ROADMAP : "l'auto-fetch écrase toujours la saisie manuelle si et seulement si il retourne une valeur non-vide") — protège contre un hoquet ponctuel de l'API qui renverrait `null` sur un match déjà scoré          | Écraser sans condition (écarté : viole la règle déjà actée) ; garder un historique séparé pour audit (écarté, over-engineering pour un outil perso, pas demandé) | actif  |
| Ce premier run remplace intégralement le round-robin fake de Phase 1 par les vraies données 2026-2027 (306 matchs) — clôt la tâche 1.1 en même temps que ça démarre Phase 6 | Effet de bord naturel et voulu : le fake n'avait plus d'utilité une fois une vraie source branchée, pas de valeur à le préserver                                                                                                                                                        | Merge partiel fake/réel (écarté, aucun intérêt — les appariements du round-robin fake ne correspondent de toute façon pas aux vrais tirages)                     | actif  |
| Scope de cette itération : script lancé manuellement (`pnpm sync-matches`), pas de cron/push/heartbeat                                                                      | Choix explicite de Baptiste : valider le script à la main avant d'investir dans l'automatisation (cron PM2 RPi, commit+push auto, heartbeat Uptime Kuma, catégorisation fine des erreurs `NetworkError`/`HttpError`/`StructureError`) — ces items restent non cochés au ROADMAP Phase 6 | Pipeline complet directement (écarté par Baptiste — trop de risque à automatiser un script encore jamais exécuté en conditions réelles)                          | actif  |
